# MP3 Curator — 취향 한 줄로 듣고 싶은 곡들을 자기 디바이스에

## Problem Statement
**HMW — "뭘 듣지?"부터 "내 폴더에 mp3가 떨어져 있다"까지의 거리를, 사용자가 검색·선택·다운로드를 반복하지 않고 한 번에 좁힐 수 있을까?**

## Recommended Direction

**Direction B (취향 → 일괄 MP3) 메인, Direction A (단일 URL → MP3) 부수.**

본인용 personal tool이라는 결정이 모든 곁가지(인증·결제·라이브러리·플레이어)를 잘라낸다. 남는 본질은 두 가지 진입점인데, 둘은 다른 product다:

- **A. URL→MP3** 단독으로는 vitamin이다. cobalt.tools가 이미 더 빠르고 공짜다. 학습 가치만 남는다.
- **B. 취향→일괄 MP3** 는 painkiller가 될 수 있다. "여성 20대 댄스곡 최신" 같은 한 문장과 원하는 곡 수 N을 입력하면, LLM이 곡 리스트를 만들고, 각 곡을 YouTube에서 검색·매칭하고, 일괄로 mp3가 폴더에 떨어진다. 이 흐름은 cobalt에도 Spotify에도 없다.

A는 B의 동일한 yt-dlp 백엔드에 URL 단축 진입점만 얹으면 그대로 떨어진다. 그래서 메인 product 정체성은 B이고, A는 같은 화면 하단의 한 줄 입력으로 흡수된다.

## Key Assumptions to Validate

가장 위험한 가정 = M1, M3. **UI 한 줄 만들기 전에** 노트북 스크립트로 먼저 검증.

- [ ] **M1 (Must)**: LLM이 만든 곡명을 YouTube에서 검색했을 때 상위 결과가 "사용자가 기대한 원본/공식 영상"과 일치한다.
  - **검증**: 1시간 probe — 취향 쿼리 5개 × LLM이 만든 곡 10개 × YouTube 검색 상위 1개 = 50개 매칭. 직접 눈으로 검수. **70% 미만이면 Direction B를 접거나 "LLM 곡 리스트를 사용자가 편집하는 모드"로 재설계**. probe 시 N=10과 N=30 둘 다 측정해 정확도가 곡 수에 따라 어떻게 무너지는지 확인.
- [ ] **M2 (Must)**: yt-dlp가 안정적으로 mp3만 추출 (오늘 기준 가능, 다만 YouTube의 클라이언트 차단에 주기적으로 깨지는 문제 — 운영 시 업데이트 필요).
  - **검증**: probe 단계에서 같이 확인.
- [ ] **M3 (Must)**: LLM이 자연어 취향에서 "환각 없는" 실제 곡 리스트를 만든다.
  - **검증**: 같은 probe에서 노출. 환각 곡(존재하지 않는 곡명) 비율 확인.
- [ ] **S1 (Should)**: N곡 일괄 변환이 곡당 약 30초 이내로 끝난다 (직렬 처리 가정. 병렬화는 yt-dlp 동시 요청 안정성 검증 후).

## MVP Scope

**한 문장의 성공 기준**:
> 취향 한 문장과 곡 수 N(1–50)을 입력하면, 내가 의도한 곡이 70% 이상 정확하게 N개의 mp3로 폴더에 들어와 있다. (곡당 약 30초 이내)

**포함**:
- 자연어 취향 입력 한 줄 (예: "여성 20대가 좋아하는 최신 댄스곡")
- 곡 수 입력 (기본 10, 1–50 범위에서 사용자 변경 가능)
- LLM 호출 → 곡 리스트(아티스트+곡명) N개 생성
- 각 곡 YouTube 검색 → 상위 1개 매칭
- yt-dlp + ffmpeg 백엔드로 일괄 mp3 변환
- 진행 상황 표시 (큐 / 곡별 상태)
- 동일 화면 하단에 "URL 직접 입력" 한 줄 (A 모드, 같은 백엔드 재사용)
- 다운로드 폴더: 로컬 디스크 (본인용이라 OK)

**빠짐 (Not Doing 참조)**.

## Not Doing (and Why)

- **계정·인증·결제** — 본인용 personal tool. 사용자가 1명이라 불필요.
- **인앱 플레이어·라이브러리 관리** — OS의 mp3 플레이어가 한다.
- **비디오 다운로드** — 오디오만이 목적.
- **ID3 태그 자동화 (자동 제목·아티스트·앨범아트)** — S2지만 MVP 이후. M1이 더 위험.
- **비슷한 곡 추천 / 데일리 자동 큐레이션** — 별도 product. 본인용 MVP에는 oversized.
- **YouTube 좋아요 import / OAuth** — 입력은 자연어 한 줄로 충분.
- **음질 옵션 (128/192/320)** — 기본값 하나로 충분. MVP 이후.
- **곡 수 50개 초과** — LLM 환각·중복·검색 매칭 실패가 누적되는 구간. 본인용 시나리오에서 한 번에 50곡 이상 필요한 경우는 드물다.
- **공개 호스팅** — 본인용이라 로컬에서 `bun dev`로 충분. ToS 리스크 회피.
- **영상 클립 / 챕터 추출** — 다른 painkiller. 흥미롭지만 이 product는 아님.

## Open Questions

- **LLM 호출**: OpenAI? Anthropic? 로컬 (Ollama)? 본인용이고 비용 민감도 낮으면 Claude API가 가장 곡 큐레이션이 정확할 가능성. **probe 단계에서 비교 권장.**
- **YouTube 검색 매칭**: YouTube Data API v3 (할당량 제한) vs `youtube-search-api` 같은 비공식 라이브러리. 본인용이라 비공식 라이브러리도 허용 범위.
- **변환 백엔드**: Next.js 서버 액션에서 yt-dlp + ffmpeg subprocess. Windows라 yt-dlp.exe·ffmpeg.exe 의존. → spec 단계에서 확정.
- **probe 결과가 70% 미만이면**: "LLM이 만든 리스트를 사용자가 편집 후 변환" 모드로 가는가, 아니면 product를 접는가?
